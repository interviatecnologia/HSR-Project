
DOCUMENTO DE API NÃO AGENTE Iniciado: 2008-07-24 Atualizado: 2024-08-24

Este documento descreve as funções de uma API (Programação de Aplicativos).
Interface) para todas as funções NÃO diretamente relacionadas à tela do Agente VICIDIAL.
Esta funcionalidade será bastante limitada no início e será desenvolvida à medida que
funções críticas são identificadas e programadas nele.

Há também um script de API do agente, para mais informações sobre isso, leia
o documento AGENT_API.txt.

OBSERVAÇÃO: A API pode usar HTTPS se o seu servidor web estiver configurado para usar HTTPS.
	Por exemplo, todas as contas VICIhost podem usar HTTPS para funções de API


Funções da API:
versão - mostra a versão e a compilação da API, juntamente com a data/hora
sounds_list - gera uma lista de arquivos de áudio do armazenamento de áudio
moh_list - gera uma lista de músicas em classes de espera no sistema
vm_list - exibe uma lista de caixas de correio de voz no sistema
blind_monitor - chama telefones definidos pelo usuário e os coloca em sessão como monitor cego
add_lead - adiciona um novo lead à tabela vicidial_list com vários campos e opções
update_lead - atualiza informações de leads em vicidial_list e tabela personalizada associada
batch_update_lead - atualiza vários leads em vicidial_list com um único conjunto de valores de campo (opções de campo limitadas)
agent_ingroup_info - mostra informações de discagem automática de grupo e de saída para o agente conectado
recording_lookup - procura gravações com base no usuário e na data ou lead_id
did_log_export - exporta todas as chamadas de entrada para um DID por um dia
agent_stats_export - exporta estatísticas do agente para um período de tempo definido
add_user - adiciona um usuário ao sistema
update_user - atualiza algumas configurações do usuário no sistema
copy_user - copia um usuário existente no sistema para um novo ID de usuário e nome
add_phone - adiciona um telefone ao sistema
update_phone - atualiza ou exclui um registro telefônico existente no sistema
add_phone_alias - adiciona um registro de alias de telefone ao sistema
update_phone_alias - atualiza ou exclui um registro de alias de telefone existente no sistema
add_list - adiciona uma lista ao sistema
update_list - atualiza as configurações da lista no sistema, redefine os leads na lista, exclui a lista
list_info - informações resumidas sobre uma lista
list_custom_fields - mostra os campos personalizados que estão em uma lista ou todas as listas
add_group_alias - adiciona alias de grupo ao sistema
user_group_status - status em tempo real de um ou mais grupos de usuários
in_group_status - status em tempo real de um ou mais grupos
agent_status - status em tempo real de um usuário
callid_info - informações sobre uma chamada com base no caller_code ou ID da chamada
lead_field_info - extrai o valor de um campo de um lead
lead_callback_info - gera dados de retorno de chamada agendados para um lead específico
lead_all_info - exibe todos os dados de um lead (opcionalmente incluindo campos personalizados)
lead_status_search - exibe todos os valores de campo de todos os leads que correspondem ao status e à data da chamada na solicitação
lead_search - pesquisa leads na tabela vicidial_list por números de telefone
ccc_lead_info - gera dados de lead para chamada de comunicação entre clusters
server_refresh - força uma atualização do arquivo conf em todos os servidores telco no cluster
check_phone_number - permite que você verifique se um número de telefone é válido e discável
logged_in_agents - lista de agentes que estão logados no sistema
update_campaign - atualiza as configurações da campanha no sistema
add_did - adiciona novas entradas DID de entrada ao sistema
copy_did - adiciona novas entradas DID de entrada ao sistema, copiadas de uma entrada DID existente
update_did - atualiza informações de DID de entrada no sistema
phone_number_log - exporta lista de chamadas feitas para um ou mais números de telefone
call_status_stats - relatório sobre o número de chamadas feitas por campanha e grupo interno, com detalhamento por hora e status
call_dispo_report - relatório de detalhamento de disposição de chamadas
agent_campaigns - procura campanhas/grupos permitidos para um usuário específico
update_cid_group_entry - atualiza entradas do grupo CID
add_dnc_phone - adiciona um número de telefone a uma das listas DNC
delete_dnc_phone - remove um número de telefone de uma das listas DNC
add_fpg_phone - adiciona um número de telefone a um grupo de telefone de filtro
delete_fpg_phone - remove um número de telefone de um grupo de telefones de filtro
campaigns_list - exibe informações sobre todas as campanhas no sistema
hopper_list - exibe informações sobre leads no hopper para uma campanha
update_alt_url - atualiza/adiciona/exibe entradas de URL de chamada de dispo alternativas para uma campanha
update_presets - atualiza/adiciona/exibe/exclui entradas de predefinições de campanha
lead_dearchive - move um lead do arquivo para a tabela de leads ativos
update_remote_agent - atualiza algumas configurações do agente remoto no sistema
user_details - exibe informações sobre um usuário

As funções da API usam a variável 'function'

NOTA: Assim como a API do agente, a API sem agente requer o usuário e a senha de um
conta vicidial_users com API válida para executar ações.

Exemplo de resposta se as permissões do usuário não autorizarem a função tentada:
ERRO: O USUÁRIO auth NÃO TEM PERMISSÃO PARA USAR ESTA FUNÇÃO - 6666|add_lead

versão - mostra a versão e a compilação da API, juntamente com a data/hora e o fuso horário

Exemplo de sequências de URL para chamadas de API:
http://server/vicidial/non_agent_api.php?function=versão

Respostas de exemplo:
VERSÃO: 2.4-34|BUILD: 110424-0854|DATA: 2011-05-29 12:19:22|ÉPOCA: 1306685962|DST: 1|TZ: -5|TZNOW: -4|

sounds_list - gera uma lista de arquivos de áudio do armazenamento de áudio

NOTA: o usuário da API para esta função deve ter o user_level definido como 7 ou superior

CAMPOS OPCIONAIS-
formato - formato da saída (guia, link, quadro de seleção)
estágio - como classificar a saída (data, tamanho, nome)
comentários - nome do campo a ser preenchido


Exemplo de sequências de URL para chamadas de API:
http://server/vicidial/non_agent_api.php?source=test&user=6666&pass=1234&function=sounds_list&format=selectframe&comments=fieldname&stage=date

Respostas de exemplo:
<o sucesso é inferido pela saída da lista de arquivos de áudio>

ERRO: sounds_list O USUÁRIO NÃO TEM PERMISSÃO PARA VER A LISTA DE SONS
ERRO: o diretório da web da loja de áudio não existe: |6666|sounds_list|/srv/www/htdocs/1234|
ERRO: sounds_list O CONTROLE DE SOM CENTRAL NÃO ESTÁ ATIVO

blind_monitor - chama telefones definidos pelo usuário e os coloca em sessão como monitor cego

NOTA: o usuário da API para esta função deve ter o user_level definido como 7 ou superior

CAMPOS OBRIGATÓRIOS-
phone_login - alfanumérico, sem espaços ou caracteres especiais permitidos
session_id - deve ser todo número, 7 dígitos
server_ip - deve ser composto apenas por números e pontos, máximo de 15 caracteres
fonte - descrição do que originou a chamada da API (máximo de 20 caracteres)
estágio - MONITOR, BARGE (ou BARGESWAP) ou HIJACK, o padrão é MONITOR
			 (A opção HIJACK não está funcional no momento)


Exemplo de sequências de URL para chamadas de API:
http://server/vicidial/non_agent_api.php?source=test&user=6666&pass=1234&function=blind_monitor&phone_login=350a&session_id=8600051&server_ip=10.10.10.16&stage=MONITOR

Respostas de exemplo:
SUCESSO: blind_monitor FOI LANÇADO - 350a|010*010*010*017*350|8600051

ERRO: blind_monitor LOGIN DE TELEFONE INVÁLIDO - 350q

ERRO: blind_monitor ID DE SESSÃO INVÁLIDO - 8602051

ERRO: O USUÁRIO DO BLIND_MONITOR NÃO TEM PERMISSÃO PARA O BLIND MONITOR - 6666|0

ERRO: NENHUMA FUNÇÃO ESPECIFICADA

agent_campaigns - procura campanhas/grupos permitidos para um usuário específico

NOTA: o usuário da API para esta função deve ter o user_level definido como 8 ou superior e "visualizar relatórios" e "modificar usuários" habilitados

CAMPOS OBRIGATÓRIOS-
agent_user - 2-20 caracteres
fonte - descrição do que originou a chamada da API (máximo de 20 caracteres)

CAMPOS OPCIONAIS-
campaign_id - 2-8 caracteres
ignore_agentdirect - S ou N, o padrão é N. Excluir AGENTDIRECT em grupos dos resultados ou não

CAMPOS DE CONFIGURAÇÕES-
stage - o formato dos dados exportados: csv, tab, pipe (padrão)
cabeçalho - incluir um cabeçalho (SIM) ou não (NÃO). Isso é opcional, o padrão é não incluir um cabeçalho

Exemplo de sequências de URL para chamadas de API:
http://server/vicidial/non_agent_api.php?source=test&function=agent_campaigns&user=6666&pass=1234&agent_user=1000
http://server/vicidial/non_agent_api.php?source=test&function=agent_campaigns&campaign_id=TESTCAMP&ignore_agentdirect=Y&user=6666&pass=1234&agent_user=1000&stage=csv

Respostas de exemplo:
ERRO: agent_campaigns O USUÁRIO NÃO TEM PERMISSÃO PARA OBTER INFORMAÇÕES DO AGENTE - 6666|0
ERRO: agent_campaigns ID DE USUÁRIO INVÁLIDO - 1255|6666
ERRO: agent_campaigns USUÁRIO DO AGENTE NÃO EXISTE - 1000|6666
ERRO: agent_campaigns ESTE USUÁRIO AGENTE NÃO TEM CAMPANHAS DISPONÍVEIS - 1000|6666

Uma resposta SUCCESS não mostrará "SUCCESS", mas apenas imprimirá os resultados no seguinte formato:
usuário|lista_de_campanhas_permitidas|lista_de_grupos_permitidos
1000|TESTCAMP-TESTE123-TESTE987|TESTE_EM-TESTE_EM2-TESTE_EM3-TESTE_EM4-LINHA_DE_VENDAS

			Aqui estão as opções duplicate_check:

				DUPTITLEALTPHONECAMP - verifique se há títulos duplicados e alt_phone em todas as listas para a campanha desta lista
				DUPTITLEALTPHONESYS - verifique se há títulos duplicados e alt_phone em todo o sistema
				DUPNAMEPHONELIST - verifique se há nomes, sobrenomes e números de telefone duplicados na mesma lista
				DUPNAMEPHONECAMP - verifique se há nomes, sobrenomes e números de telefone duplicados em todas as listas da campanha desta lista
				DUPNAMEPHONESYS - verifique se há nomes, sobrenomes e números de telefone duplicados em todo o sistema
				 " 1/2/3/7/14/15/21/28/30/60/90/180/360DAY - Adicionado a uma das verificações duplicadas acima (ou seja, "DUPSYS90DAY"), verifica apenas os leads carregados nos últimos 90 dias

update_lead - atualiza informações de leads nas tabelas vicidial_list e custom_

NOTA: o usuário da API para esta função deve ter modify_leads definido como 1 e user_level
      deve ser definido como 8 ou superior

CAMPOS OBRIGATÓRIOS-
lead_id - deve ser composto apenas por números, de 1 a 9 dígitos, não é necessário se usar vendor_lead_code ou phone_number
vendor_lead_code - pode ser usado em vez de lead_id para corresponder a leads
phone_number - pode ser usado em vez de lead_id ou vendor_lead_code para corresponder a leads
fonte - descrição do que originou a chamada da API (máximo de 20 caracteres)

CAMPOS DE CONFIGURAÇÕES-
search_method - Você pode combinar as 3 opções a seguir neste campo para pesquisar os parâmetros desejados:
				LEAD_ID, tentará encontrar uma correspondência com o lead_id
				VENDOR_LEAD_CODE, tentará encontrar uma correspondência com o vendor_lead_code
				PHONE_NUMBER, tentará encontrar uma correspondência com o phone_number
			  Por exemplo, para pesquisar lead_id e vendor_lead_code: "&search_method=LEAD_ID_VENDOR_LEAD_CODE"
			  A ordem de pesquisa NÃO é preservada, o ID do lead é sempre o primeiro, o código do lead do fornecedor é o segundo
			  e o número de telefone é o último. O padrão é "LEAD_ID"
search_location - Onde verificar registros no sistema, pode selecionar apenas um (o padrão é SYSTEM):
				LISTA - verificar se há chumbo na mesma lista
				CAMPANHA - verificar se há leads em todas as listas para a campanha desta lista
				SISTEMA - verificar se há chumbo em todo o sistema
			  Se nenhum list_id for definido, o search_location será assumido como SYSTEM
insert_if_not_found - S ou N, tentará inserir como um NOVO lead se nenhuma correspondência for encontrada, o padrão é N.
			A inserção exigirá phone_code, phone_number e list_id. lead_id será ignorado.
			A maioria das opções add_lead que não estão disponíveis se você usar esta configuração nesta função
registros - número de registros a serem atualizados se mais de 1 for encontrado (o padrão é '1' [lead carregado mais recentemente])
custom_fields - Y ou N, o padrão é N. Define se a API aceitará dados de campos personalizados ao atualizar leads na tabela vicidial_list
			Para que os campos personalizados sejam atualizados, basta adicionar o rótulo do campo como uma variável à string do URL
			Por exemplo, se o field_label for "favorite_color", você adicionaria "&favorite_color=blue"
no_update - S ou N, Definir como S não executará nenhuma atualização, mas apenas informará
			se existir um lead que corresponda aos critérios de pesquisa, o padrão é N.
delete_lead - S ou N, definir como S excluirá o lead da tabela vicidial_list, o padrão é N.
delete_cf_data - S ou N, definir como S excluirá os dados dos campos personalizados para esse lead e redefinirá o entry_list_id, o padrão é N.
reset_lead - S ou N, definir como S redefinirá o sinalizador chamado desde a última reinicialização do lead, o padrão é N.
retorno de chamada - Y, N ou REMOVE, o padrão é N. Defina este lead como um retorno de chamada agendado. REMOVE excluirá a entrada de retorno de chamada agendada
callback_status - 1-6 caracteres, status de retorno de chamada a ser usado, o padrão é CALLBK (o status vicidial_list será definido como CBHOLD para bloquear)
callback_datetime - AAAA-MM-DD+HH:MM:SS, data e hora do retorno de chamada agendado. OBRIGATÓRIO se o retorno de chamada estiver definido.
                        'NOW' pode ser usado para data e hora atuais. 'xDAYS' também pode ser usado onde 'x' é substituído por um número de dias no futuro
callback_type - USERONLY ou ANYONE, o padrão é ANYONE
callback_user - ID do usuário ao qual o retorno de chamada USERONLY é atribuído
callback_comments - Comentários opcionais para aparecer quando o retorno de chamada for chamado de volta
update_phone_number - S ou N, configuração opcional para atualizar o campo phone_number, o padrão é N.
add_to_hopper - S ou N, o padrão é N
remove_from_hopper - S ou N, o padrão é N
hopper_priority - 99 a -99, quanto maior o número, maior a prioridade, o padrão é 0
hopper_local_call_time_check - S ou N, o padrão é N. Valide o tempo de chamada local e/ou o tempo de chamada do estado antes de inserir o cabo no funil
list_exists_check - S ou N, o padrão é N. Se o list_id_field não for uma lista definida no sistema, ocorrerá ERRO e o lead não será atualizado.
archived_lead - S ou N, definir como S consultará apenas os leads vicidial_list_archive para pesquisa, atualizações e detalhes de leads; o padrão é N.

CAMPOS EDITÁVEIS-
user_field - 1-20 caracteres, isso atualiza o campo 'user' na tabela vicidial_list
list_id_field - 3-12 dígitos, isso atualiza o campo 'list_id' na tabela vicidial_list
status - 1-6 caracteres, sem pontuação ou espaços
vendor_lead_code - 1-20 caracteres
source_id - 1-50 caracteres
gmt_offset_now - substituído pela pesquisa automática da parte phone_code e area_code do número de telefone, se aplicável
título - 1-4 caracteres
first_name - 1-30 caracteres
middle_initial - 1 caractere
sobrenome - 1-30 caracteres
endereço1 - 1-100 caracteres
endereço2 - 1-100 caracteres
endereço3 - 1-100 caracteres
cidade - 1-50 caracteres
estado - 2 caracteres
província - 1-50 caracteres
postal_code - 1-10 caracteres
country_code - 3 caracteres

data_de_nascimento - AAAA-MM-DD
alt_phone - 1-12 caracteres
e-mail - 1-70 caracteres
frase_de_segurança - 1-100 caracteres
comentários - 1-255 caracteres

called_count - apenas dígitos, o número de tentativas de discagem do lead

entry_list_id - AVISO! USE SOMENTE SE VOCÊ SABE O QUE ESTÁ FAZENDO, PODE QUEBRAR CAMPOS PERSONALIZADOS! (deve ser tudo números, 3-12 dígitos, não funcionará se custom_fields estiver definido como Y)
force_entry_list_id - AVISO! USE SOMENTE SE VOCÊ SABE O QUE ESTÁ FAZENDO, PODE QUEBRAR CAMPOS PERSONALIZADOS! (deve ser tudo números, 3-12 dígitos, substituirá entry_list_id para este valor em todas as consultas de campos personalizados executadas por este comando)
NOTAS:
 - para definir um campo como vazio(''), defina-o como --BLANK--, ou seja, "&province=--BLANK--"
 - não use caracteres especiais como apóstrofos, aspas duplas ou e-feras

check_phone_number - permite que você verifique se um número de telefone é válido e discável

NOTA: o usuário da API para esta função deve ter o user_level definido como 8 ou superior

CAMPOS OBRIGATÓRIOS-
phone_number - deve ser todos números, 6-16 dígitos
phone_code - deve ser composto apenas por números, de 1 a 4 dígitos, o padrão é 1 se não for definido
local_call_time - deve ser um ID de tempo de chamada válido no sistema

CAMPOS OPCIONAIS-
postal_code - deve ter 5 dígitos. Funciona apenas para códigos postais dos EUA, necessário para POSTAL tz_method
estado - deve ter 2 letras. Necessário somente se as regras de tempo de chamada de estado estiverem habilitadas
proprietário - deve ter de 2 a 5 letras. Necessário somente se estiver usando TZCODE tz_method

CAMPOS DE CONFIGURAÇÕES-
dnc_check - Y, N ou AREACODE, o padrão é N
campaign_dnc_check - S, N ou AREACODE, o padrão é N
campaign_id - ID de campanha de 2 a 8 caracteres, necessário se estiver usando campaign_dnc_check
usacan_prefix_check - S ou N, o padrão é N. Verifique se há um 4º dígito válido para números de telefone dos EUA e Canadá (não pode ser 0 ou 1)
usacan_areacode_check - S ou N, o padrão é N. Verifique se há um código de área válido para números de telefone dos EUA e Canadá
nanpa_ac_prefix_check - Y ou N, o padrão é N. Verifique se há um código de área e prefixo NANPA válidos, se houver dados NANPA opcionais no sistema
tz_method - <empty>, POSTAL, TZCODE, NANPA ou AREACODE, o padrão é <empty> que usará o código do país e AREACODE para pesquisas de fuso horário
				AREACODE depende do código de área do número de telefone
				POSTAL depende do campo postal_code
				O TZCODE depende do preenchimento do campo do proprietário com um código de fuso horário adequado
				O NANPA depende dos dados de prefixo de código de área NANPA opcionais que estão sendo carregados no seu sistema
				(NOTA: você pode fazer mais de um método ao mesmo tempo, por exemplo: "POSTALTZCODE")

CAMPOS OBRIGATÓRIOS-
did_pattern - 2-50 caracteres, deve ser um DID válido no sistema
			   NOTA: caracteres especiais no did_pattern precisam ser codificados em URL, por exemplo, o sinal de mais '+' deve ser enviado como '%2B'

CAMPOS EDITÁVEIS-

phone_code - Se a rota for IN_GROUP, este é o código do telefone, padrão '1'
delete_did - Pode ser um destes: 'Y','N' (o padrão é 'N')
